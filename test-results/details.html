DOM4: Analysis of failed tests

Introduction

Historically, W3C test suites have altogether too often been subpar in their coverage, barely scraping the surface of interoperability. In such a context, with only a few hundred simple tests, it is naturally expected that one would expect to hit the 100% pass mark.

With the more recent focus on testing however, notably through the Web Platform Tests and Test The Web Forward projects the quality of test suites has improved massively. And since we don't live in a perfect world, if one looks for trouble hard enough one will find it.

The DOM test suite, which is now 47,000 tests strong (and growing), certainly did set out to look for trouble. Given that, it seems useful to accompany the test results with this this note explaining where the failures come from.

In total, 1719 tests fail to pass in two implementations, which amounts to 3.65% of the total. The sections below provide context in which to appraise these failures.

Failures from Range.compareBoundaryPoints()

A full 1177 of the 1719 (68.5%) come from a single test file. This test file generates 8000 tests by going through a list of ranges that need comparing under a wide array of configurations. These failures break down into two large blocks:

Problematic use of detach()
The detach() method was used in the test, but it has since become deprecated (or rather, it is defined to do nothing). A fix is making its way through the test suite process.
Validation of the first argument
Many tests check that the first argument, which is an enumeration of shorts, produces the correct error when something else is provided. For these tests, several browsers fail by throwing an error of the wrong type (which boils down to issues they have with the WebIDL algorithm for conversions to numbers). This is of particularly low importance in JavaScript since the catch clause of a try cannot switch on exception type. So getting an exception in the first place (which is happening) is vitally more important than getting an exception of the right type.

Ten other files are seeing the same detach() bug (fix in progress), accounting for a further 133 failures.

Faulty tests and WebIDL argument conversion failures leading to exceptions of the wrong type do not point at any interoperability issue of concern. Eliminating these results leaves us with 379 failures (0.80% of total).

WebIDL related issues

Thirteen of the test files exhibit various degrees of WebIDL-related problems, for a total of 329 tests. These are largely in the interfaces.html tests which are auto-generated WebIDL conformance tests. In many cases, if just one bug were fixed the number of failures would fall dramatically.

Failures in other files are usually either conversion errors (e.g. failing to convert undefined to the empty string) or exception type errors.

None of these failures are particularly important and actual in-the-trenches interoperability only suffers ever so lightly from these.

Eliminating these leaves us with 50 failures (0.10%).

Event failures

Four files, with a total of 31 failures, concern Events. Two of those are for Progress Events which are defined in a separate specification (they will be moved). The rest is essentially issues with default values being applied properly to synthetically created events. While these issues would possibly cause interoperability bugs in real code, they do not demonstrate implementatibility problems and we are confident that they will get fixed.

This leaves us with 19 failures (0.04%).

Miscellaneous

What remains is a hodgepodge of small corner case problems. A number of the failures are due to checks that old, now obsolete, DOM features (e.g. CDATASection) are indeed gone; but it is not a normative requirement of the specification that they be actually removed.

A few more have to do with stringification of some attributes, and obscure failures from the diminishing returns corner.

Conclusion

Considering that:

  1. the volume of failures is low to start with;
  2. the vast majority come from single sources that do not pose major threats to interoperability;
  3. the percentage of issues that point to problems that could affect developers is so vanishingly small as to be negligible;

We contend that the specification being proposed for advancement through the process has demonstrated interoperability.
